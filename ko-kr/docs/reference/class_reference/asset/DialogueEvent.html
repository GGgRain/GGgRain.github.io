DialogueEvent

[!Info] 본 문서는 다이얼로그 이벤트의 기능과 구성에 대한 세부적인 설명을 포함합니다. 실제 사용 예시는 [[이벤트 써보기]] 문서를 참고하십시오.

다이얼로그 이벤트는 [[다이얼로그 매니저]]에 포함된 [[다이얼로그 노드]]에 붙혀 런타임에서 사용 가능한 블루프린트 로직을 담을수 있는 클래스입니다.

노드와 라이프사이클을 같이 하는 별도의 인스턴스를 가지며, 모든 로직이나 벨류가 인스턴스에 직접 기입 되기에 [[다이얼로그 인라인 커맨드]]와는 다르게 스크립트 변경으로 인한 오류 및 보안 취약점 발생 우려가 적습니다.

다이얼로그 이벤트 종류​

다이얼로그 이벤트는 두가지 종류가 있습니다.

  • 인스턴스드 다이얼로그 이벤트
  • 레벨 다이얼로그 이벤트

인스턴스드 다이얼로그 이벤트는 다이얼로그 매니저 상의 다이얼로그 노드 인스턴스에 인스턴스 형태로 부착 및 저장되는 이벤트입니다. 일반적으로 가장 많이 사용되는 형태입니다.

레벨 다이얼로그 이벤트는 말 그대로 액터로서 레벨에 배치가 가능한 형태의 이벤트입니다. 다이얼로그 매니저에서 이 이벤트를 가입시키면 그냥 그 이벤트 액터에 대한 소프트 액터 레퍼런스만을 가집니다. 월드 특정 위치, 혹은 월드 파티션등의 특수효과에 의한 영향을 주어야할 때 사용하기 좋습니다.

다이얼로그 이벤트 정의​

[!Info] 이벤트 에셋의 생성 및 사용에 관한 각 절차마다 세부적인 설명은 [[이벤트 써보기]] 문서를 참고 하십시오.

다이얼로그 이벤트 에셋을 생성 한 뒤에 그 에셋에 부여할 이벤트 로직을 작성하는 것은 다음과 같은 함수를 오버라이드 하는 것에서 시작됩니다 :

C++ 자손 클래스의 경우 : virtual void ReceiveEventTrigger(class UDialogueWidgetBase* Widget, const TMap<FName, UActorComponent*>& ConversationParticipants);

블루프린트 에셋인 경우 :

여기에 원하는 로직을 넣어주면 됩니다.

예제로 주어진 Dilation 값에 따라 게임의 타임 스케일을 조작하는 DE_SetGlobalTimeDilation에셋의 ReceiveEventTrigger 이벤트 로직입니다.

다이얼로그 이벤트 호출​

인라인 커맨드를 이용해 호출​

기본으로 제공 되는 [[다이얼로그 인라인 커맨드]] 중, DI_Event (시그니처: Event) 커맨드를 사용하여 텍스트 업데이트 도중 텍스트의 원하는 지점에서 다이얼로그 이벤트를 호출 할 수 있습니다.

![[Time.png]]

해당 커맨드를 넣은 뒤, Value 어트리뷰트에 호출하고자 하는 이벤트의 키 값을 넣어주면 해당 키에 대응하는 이벤트가 호출 됩니다.

[!Info] 자세한 내용은 [[이벤트 써보기]] 문서를 참고하십시오.

익스큐트 노드를 이용해 호출​

[[익스큐트 노드]]를 사용해서도 다이얼로그 이벤트를 호출 할 수 있습니다.

인라인 커맨드를 사용하지 않기 때문에 현지화 과정에서 액션이 누락될 걱정을 하지 않아도 된다는 장점이 있습니다. 따라서 게임 플레이 도중 반드시 실행되어야 하는 액션을 실행하기에 적합합니다.

[!Info] 자세한 내용은 [[익스큐트 노드]] 문서를 참고하십시오.

부가적 요소​

FDialogueEventStructure​

Simple Dialogue System 플러그인에서는 일반적으로 다이얼로그 이벤트의 집합을 FDialogueEventStructure라는 이름의 스트럭쳐로 묶어서 처리합니다.

이 스트럭쳐는 간단하게 인스턴스드 이벤트와 레벨 이벤트를 담을 수 있는 배열과 두 배열 내의 모든 컨디션을 일괄적으로 호출 할 수 있는 함수가 담긴 랩퍼 스트럭쳐입니다..

실제로 다이얼로그 노드들은 따로 다이얼로그 이벤트를 위한 각각의 배열을 가지는게 아니라, FDialogueEventStructure 스트럭쳐를 사용하여 이벤트를 저장합니다.

다이얼로그 노드 클래스 상에선 맵 컨테이너를 사용하여 이벤트 호출에 사용할 구분 키 스트링과 FDialogueEventStructure을 일대일로 대응하여 저장 해 두었다가, 특정 키에 대한 이벤트 호출 명령이 내려지면 그 키에 대응하는 FDialogueEventStructure의 모든 이벤트를 호츨하는 방식으로 작동합니다.

[!Info] 배열을 사용하지 않고 스트링-스트럭쳐 맵을 사용한 이유는 관성적인 부분이 큽니다.

이런 구조가 처음에 들어선 이유는 플러그인의 초기 버전에서는 인라인 커맨드라는 개념이 없었고, 모든 인라인 액션이 이벤트의 키를 텍스트 안에서 [[, ]]안에 담아서 이벤트를 호출하는 방식으로 구현되었기 때문입니다. 따라서 이벤트를 키에 따라서 구분하는 것을 필수적이었습니다.

만약 커스텀 노드를 작성 할 때 각각의 이벤트 스트럭쳐를 구분할 필요 없이 그냥 전부 저장했다가 일괄적으로 사용하고자 한다면 그냥 배열을 새로 정의해서 담아도 문제가 없을 것입니다.

실제로 [[익스큐트 노드]]는 이런 맵 구조보다는 배열을 이용해서 FDialogueEventStructure를 저장하는 것이 유리했고, 따라서 저희는 아예 기존 노드 클래스의 Events 프로퍼티를 숨기고 아예 새로운 배열 프로퍼티를 정의하여 이벤트를 저장했습니다.

![[eventstr.png]]

또한 UDialogueFunctionLibrary::TriggerEventStruct 함수를 이용해 블루프린트 상에서도 FDialogueEventStructure의 모든 이벤트를 호출 할 수 있습니다.
